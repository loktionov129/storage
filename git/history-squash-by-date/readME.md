# history-squash-by-date
## Скрипт для сквоша Git‑коммитов по периодам


Утилита на Node.js для объединения (сквоша) множества Git‑коммитов в более компактные группы по временным периодам: годам, месяцам или дням.

## Назначение

Скрипт автоматизирует процесс сквоша коммитов в Git‑репозитории, группируя их по:
* годам (`year`);
* месяцам (`month`);
* дням (`day`).

Для каждого периода создаётся один коммит с осмысленным сообщением, анализирующим тематику изменений.

## Особенности

* **Автоматическое определение тематики** — на основе ключевых слов Conventional Commits.
* **Гибкая группировка** — выбор стратегии (`year`/`month`/`day`).
* **Безопасность** — проверка чистого состояния репозитория перед началом работы.
* **Отчётность** — вывод статистики до и после сквоша.
* **Обработка ошибок** — корректное завершение при сбоях, отмена rebase.

## Установка и требования

1. Убедитесь, что установлен **Node.js** (версия 12+).
2. Сохраните скрипт как `squashHistory.js` в корне репозитория.
3. Убедитесь, что у вас есть права на выполнение операций `git rebase -i` и `git commit --amend`.

## Использование

Запустите скрипт из корня Git‑репозитория:

```bash
node squashHistory.js --strategy <period>
```

Где `<period>` — одна из стратегий:
* `year` — группировать коммиты по годам;
* `month` — группировать по месяцам;
* `day` — группировать по дням.


### Примеры команд

```bash
# Сквашить все коммиты по годам
node squashHistory.js --strategy year

# Сквашить все коммиты по месяцам
node squashHistory.js --strategy month


# Сквашить все коммиты по дням
node squashHistory.js --strategy day
```

## Важные замечания

1. **Резервное копирование**: перед запуском сделайте бэкап репозитория или создайте ветку для экспериментов.
2. **Чистое состояние**: скрипт не запустится, если есть незакоммиченные изменения или незапушенные коммиты.
3. **Интерактивный rebase**: скрипт использует `git rebase -i`, который может вызвать конфликты. В случае конфликта разрешите его вручную и продолжите rebase.
4. **Удалённые репозитории**: сквош переписывает историю. Если репозиторий уже опубликован, убедитесь, что все участники готовы к перезаписи истории. Используйте с осторожностью в командных проектах.
5. **Стратегия выбора**:
    * `year` — для очень старых репозиториев с огромной историей;
    * `month` — оптимальный баланс между компактностью и сохранением контекста;
    * `day` — для проектов с частыми мелкими коммитами, где важно сохранить дневную гранулярность.

## Устранение неполадок


* **«Not in a Git repository»** — убедитесь, что вы находитесь в корне Git‑репозитория.
* **«Uncommitted changes»** — закоммитьте или уберите все незакоммиченные изменения.
* **Ошибки rebase** — скрипт попытается отменить операцию и очистить временные файлы. При необходимости выполните `git rebase --abort` вручную.
