# history-squash-by-date

## Скрипт `squashHistory.js` — перепись истории по периодам

Скрипт на Node.js переписывает историю Git по **году**, **месяцу** или **дню** (по **дате автора** коммита в выводе `git log`), опционально добавляет в сообщение сквоша **тему** по префиксам в стиле Conventional Commits (`feat:`, `fix:` и т.д.).

Поведение **зависит от стратегии**:

| Стратегия | Как устроено |
|------------|----------------|
| **`year`** | Одна **линейная** цепочка: **ровно один новый коммит на каждый календарный год автора**, дерево (`tree`) — как у **последнего** коммита этого года в порядке обхода лога скрипта. **Merge-коммиты исчезают** (история уплощается). Реализация: `git commit-tree` + `update-ref` + `reset --hard`. |
| **`month` / `day`** | Группы обычных коммитов за период сквошатся через **`git rebase -i --rebase-merges`**. **Merge-коммиты не сквошатся** с соседями. Часть сегментов может быть **пропущена**, если в todo нет непрерывного блока `pick` (см. флаги ниже). |

Подробная спецификация: `spec.md`. Сценарии на английском: `use_cases.md`.


## Требования

* **Node.js** 12+  
* **Git** 2.x (нужны `rebase -i`, `rebase-merges`, для года — `commit-tree`)


## Запуск

1. Репозиторий должен быть **чистым** (`git status` без изменений).  
2. Для стратегии **`year`** нужна **именованная ветка** (не detached `HEAD`).  
3. Запуск из каталога внутри репозитория (обычно корень):

```bash
node ./git/history-squash-by-date/squashHistory.js --strategy year
```


## Параметры командной строки

| Параметр | Обязательный | Описание |
|----------|----------------|----------|
| `--strategy` | Да | `year` — снимок дерева на год; `month` / `day` — интерактивный rebase по периоду. |
| `--verbose` | Нет | Печать команд `git` в stderr (`[git] ...`) и больше вывода процесса. |
| `--strict-directory-renames` | Нет | Только **month/day**: не отключать обнаружение переименований каталогов при rebase (по умолчанию скрипт задаёт `merge.directoryRenames=false` и стратегию `recursive`, чтобы реже ловить конфликты «расположения файла» при merge + squash). |
| `--allow-legacy-fallback` | Нет | Только **month/day**: если при `--rebase-merges` нет **непрерывного** блока `pick` для периода, попытаться старым способом помечать `squash` по всему todo (**опасно**: возможен неверный родитель и конфликты rename/delete). По умолчанию такой сегмент **пропускается**. |

Для **`year`** флаги `--strict-directory-renames` и `--allow-legacy-fallback` **не используются** (скрипт может вывести короткое предупреждение).


## Пример: год — одна линия из N коммитов (N = число лет)

```bash
node ./git/history-squash-by-date/squashHistory.js --strategy year --verbose
```

Скрипт выведет план вида «год → короткий хэш источника, сколько коммитов в этом году в обходе лога», спросит подтверждение, затем перепишет текущую ветку. В результате `git rev-list --count HEAD` часто совпадает с **числом различных годов автора** в истории (например, 10 коммитов за 2017–2026).


## Пример: месяц или день

```bash
node ./git/history-squash-by-date/squashHistory.js --strategy month
node ./git/history-squash-by-date/squashHistory.js --strategy day --verbose
```

Итоговое число коммитов зависит от того, сколько сегментов удалось сквошнуть и сколько было пропущено из‑за топологии merge.


## Сообщения коммитов

Формат:

```text
squash: <число> commits for <период>
squash: <число> commits for <период> (<тема на английском>)
```

Период: `2026` (год), `2023-10` (месяц), `2023-10-01` (день). Тема — по наиболее частому типу в subjects за период (`feat`, `fix`, …).


## Ошибки и отмена

* Не Git-репозиторий → ошибка, выход.  
* Есть незакоммиченные изменения → ошибка, выход.  
* Неверная стратегия → ошибка, список `year`, `month`, `day`.  
* **`year`** на detached `HEAD` → ошибка (нужна ветка).  
* На вопрос подтверждения ответ **`n`** — выход без изменений.  
* Если застряли на **интерактивном rebase** (month/day): **`git rebase --abort`**.

Для **month/day** при «пустом» шаге squash (например, добавили файл и удалили в одном сквошe) скрипт может автоматически выполнить **`git commit --amend --allow-empty --no-edit`** и продолжить **`git rebase --continue`** (распознавание текста ошибки Git с переносами строк).


## Безопасность и ограничения

* Перепись истории **необратима** для уже опубликованных веток без согласования с командой; при необходимости сделайте резервную ветку: `git branch backup-before-squash`.  
* Скрипт рассчитан на **локальную** работу; push не выполняет.  
* **`year`:** не сохраняет merge-коммиты; выбор дерева привязан к **дате автора** и порядку `git log --reverse` в скрипте, а не к «только first-parent».  
* **`month` / `day`:** merge-коммиты не объединяются с обычными коммитами в один squash.  
* Коммиты с непарсируемой датой автора в обходе не учитываются.
